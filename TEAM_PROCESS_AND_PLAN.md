# Процесс, команда и план работ по сайту DreamHouse

> Документ описывает: (1) команду агентов с ролями и сильными сторонами,
> (2) процесс «правка → ревью руководителя → QA → отчёт», (3) текущий
> план работ. Источник истины по дизайну детальной карточки — мобильное
> Compose-приложение в этом же репозитории.

---

## 1. Команда (роли агентов)

Работаем по 5-ролевой модели (см. memory: 5-Role Team Workflow).
Каждый агент — отдельный subagent с чёткой зоной и сильной стороной.

### 🎯 Руководитель / Tech Lead (агент-ревьюер) — «ОТЧЁТ ПЕРЕД НИМ»
- **Тип:** `feature-dev:code-reviewer`
- **Сильная сторона:** находит баги, логические ошибки, нарушения
  конвенций; confidence-based фильтрация (репортит только важное).
- **Когда:** ПОСЛЕ каждой пачки правок. Получает дифф + контекст
  задачи, выносит вердикт: APPROVE / CHANGES REQUESTED со списком.
- **Перед ним отчитываемся:** ни одна правка не считается готовой,
  пока Tech Lead не дал APPROVE и не пройден QA-чеклист.

### 🔍 Исследователь архитектуры — «карта местности»
- **Тип:** `Explore` / `feature-dev:code-explorer`
- **Сильная сторона:** быстрый широкий обход кодовой базы, трассировка
  путей данных, поиск зависимостей и точек входа, без правок.
- **Когда:** перед крупными изменениями — даёт точную карту: что где
  лежит, что от чего зависит, что сломается.

### 🏗️ Архитектор — «как правильно делать»
- **Тип:** `feature-dev:code-architect`
- **Сильная сторона:** проектирует решение по существующим паттернам
  проекта, выдаёт чёткий план: файлы, компоненты, последовательность.
- **Когда:** для нетривиальных фич перед реализацией.

### 👷 Разработчики (исполнители) — делегируемые задачи
- **Тип:** `general-purpose` (по 1 на независимый блок, параллельно)
- **Сильная сторона:** изолированная реализация одного компонента по
  жёсткому ТЗ с самопроверкой tsc/lint.
- **Когда:** реализация. Один агент = один компонент/файл-набор.
  Запускаются параллельно когда задачи независимы.

### ✅ QA — «проверка после правок»
- **Роль исполняется оркестратором** (главный ассистент) по чеклисту
  ниже + дополнительно агентом-ревьюером.
- **Сильная сторона:** воспроизводимые проверки: tsc, lint, build,
  smoke (curl), регрессия ссылок, dev-лог на ошибки.

### Оркестратор (главный ассистент)
Координирует: пишет ТЗ агентам, собирает результат, гоняет QA, носит
дифф Руководителю, ведёт TodoWrite, коммитит, отчитывается пользователю.

## 2. Процесс (обязательный для каждой задачи)

```
1. ИССЛЕДОВАНИЕ   → Исследователь даёт карту (если задача нетривиальна)
2. ПЛАН           → Архитектор/оркестратор фиксируют план в .md
3. ДЕКОМПОЗИЦИЯ   → независимые блоки → параллельные Разработчики
4. РЕАЛИЗАЦИЯ     → каждый агент с самопроверкой tsc/lint
5. СБОРКА         → оркестратор собирает, чинит интеграцию
6. QA             → tsc + lint + build-compile + smoke + регрессия
7. РЕВЬЮ          → Руководитель (code-reviewer) смотрит дифф:
                    APPROVE → дальше; CHANGES → назад к п.4
8. КОММИТ         → один логический коммит, осмысленное сообщение
9. ОТЧЁТ          → пользователю: что/почему/QA-результат/риски
```

**Правило:** правка НЕ готова без (а) APPROVE Руководителя и
(б) зелёного QA-чеклиста. Никаких «и так сойдёт».

### QA-чеклист (после каждой пачки)
- [ ] `npx tsc --noEmit` — нет НОВЫХ ошибок по затронутым файлам
- [ ] `npx next lint --dir <зона>` — чисто
- [ ] dev-лог: `✓ Compiled` без ошибок, нет runtime/Module-not-found
- [ ] smoke: затронутые роуты возвращают 200
- [ ] регрессия: ссылки/соседние страницы не сломаны
- [ ] функционал сохранён (клики, модалки, redux, навигация)
- [ ] предсуществующие чужие ошибки (Header/leads/map) — НЕ трогаем,
      но фиксируем что они не наши и не растут

## 3. Архитектура сайта (краткая карта — дополняется аудитом)

- **Стек:** Next.js 15.5.9 (App Router) + React 19 + TS + Redux Toolkit
  + Tailwind v4. Шрифты Stetica/Inter/Manrope (next/font/local).
- **Репозиторий сайта:** `dreamhouse-site/` — ОТДЕЛЬНЫЙ git-репо
  (не часть монорепо `DreamHouse2`). Ветка `main`,
  origin = `github-kelemetovrasid:kelemetovrasid/dreamhouse-new-sait.git`.
- **API:** внешний бэкенд `https://api.dreamhouse05.com/api`.
- **Дизайн-токены:** `app/globals.css` — `var(--*)`, 1:1 совпадают
  с мобильным приложением `base/.../design/tokens/DreamColors.kt`.
- **Ключевые маршруты:** `/` (главная), `/card/[id]` (детальная —
  pixel-perfect порт мобильного экрана), `/sale`, `/favorite`,
  `/map`, `/developers`, `/chat`, `/profile`.
- **Карта:** `react-yandex-maps-fork` (Яндекс) — ПРОБЛЕМА, см. §4.
  `maplibre-gl` УЖЕ установлен (юзается в незакоммиченном
  `app/map/components/MapCanvas.tsx`).

## 4. Проблема карты (диагноз) и план фикса

### Диагноз (подтверждён curl + консолью браузера)
- `GET https://api-maps.yandex.ru/2.1/...` → `net::ERR_CONNECTION_CLOSED`
- `curl https://api-maps.yandex.ru` → HTTP 000 (exit 35, SSL fail)
- `curl https://yandex.ru/maps` → HTTP 000
- **Вывод:** Яндекс полностью недоступен из этой сети/региона.
  Это инфраструктурная блокировка, НЕ баг кода. Встроенный
  `react-yandex-maps` не может отрисоваться нигде (карточка, /map).

### Решение (выбрано пользователем): OpenStreetMap
- Заменить Яндекс-карту в карточке на **MapLibre GL + OSM-тайлы**
  (`maplibre-gl` уже в node_modules; OSM raster tiles
  `https://tile.openstreetmap.org/{z}/{x}/{y}.png` — без ключа,
  без блокировок, работает везде).
- Новый компонент `OsmMap.tsx` (client-only, ssr:false) с маркером
  по координатам объекта.
- `MapPreviewApp` использует OsmMap когда есть координаты; когда
  координат нет (как card 46) — оставить плашку с адресом.
- Кнопка «На карте»: вести на универсальный gmaps/2gis/yandex —
  откроется у тех юзеров, у кого доступ есть. Базово — Google Maps
  (`https://www.google.com/maps?q=` или `/dir/?api=1&destination=`)
  как наиболее доступный для маршрута; адрес/координаты.
- `MapWidget.tsx` (старый Яндекс) — оставить, но в карточке не
  использовать. `/map` (чужая незакоммиченная работа) — НЕ трогать.

### Файлы
- Создать: `app/card/[id]/lib/OsmMap.tsx`
- Изменить: `app/card/[id]/lib/MapPreviewApp.tsx`
- Не трогать: `MapWidget.tsx`, `app/map/*`, Redux, API.
